# ENUM vs 별도 테이블 — 언제 뭘 써야 하나

- 날짜: 2026-04-04

## 상황

Tarae 프로젝트 스키마 설계 중 category, status, region 등 "고정된 값 목록"을 ENUM으로 할지 별도 테이블로 할지 결정해야 했다.

## 핵심 내용

처음엔 전부 ENUM으로 했다가 category만 별도 테이블로 전환했다. 판단 기준:

| 기준 | ENUM | 별도 테이블 |
|------|------|------------|
| 값이 바뀔 가능성 | 거의 없음 | 추가/수정 가능성 있음 |
| 메타데이터 필요 | 없음 (값만 필요) | 있음 (라벨, 정렬순서, 아이콘 등) |
| API로 목록 제공 | 코드에 하드코딩 | `GET /categories` 한 줄이면 끝 |
| 스키마 변경 비용 | `ALTER TABLE` 필요 | row INSERT 한 줄 |

**category → 테이블:** 프론트 드롭다운에 목록을 내려줘야 하고, 한글 라벨/정렬순서가 필요.
**status(open/closed) → ENUM:** 2개뿐이고 메타데이터 불필요.
**region → VARCHAR:** 고정 값이지만 프론트 드롭다운으로 일관성 보장. 테이블까지는 불필요.

## 왜 중요한가

- "고정 값이니까 ENUM" 이라고 기계적으로 판단하면 나중에 `ALTER TABLE`을 치는 순간이 온다
- 반대로 전부 테이블로 만들면 불필요한 JOIN만 늘어남
- **"이 값 목록을 API로 제공해야 하나?"**가 가장 실용적인 판단 기준
